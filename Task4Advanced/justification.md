# Обоснование событийного подхода — «Будущее 2.0»

---

## Проблемы текущей архитектуры (Camel ESB + DWH)

### 1. Жёсткая связанность через ESB

Apache Camel выступает синхронным посредником: каждый сервис вызывает другой через шину и ждёт ответа. При добавлении нового домена (Фарма, МедТех) нужно менять маршруты в Camel — любое изменение затрагивает всех участников. Это классическое «spaghetti integration»: N сервисов дают N×(N-1)/2 потенциальных зависимостей.

### 2. DWH как единая точка бизнес-логики

SQL Server 2008 хранит не только данные, но и бизнес-правила в хранимых процедурах. Финтех-логика, ИИ-интеграция и аналитика перемешаны в одной БД. Изменение правила для одного домена требует координации со всеми командами, которые работают с тем же DWH.

### 3. Batch-аналитика с многочасовой задержкой

Power BI строит отчёты напрямую из DWH через тяжёлые SQL-запросы. При объёме данных в сотни ТБ построение сложных отчётов занимает часы. Near-real-time аналитика невозможна архитектурно.

### 4. Невозможность независимого масштабирования

Рост нагрузки на финтех-сервис → нагрузка на весь DWH. Пиковые медицинские записи тормозят аналитику. Ресурсы не изолированы по доменам.

### 5. Блокирующий time-to-market

Подключение нового источника данных или нового потребителя требует изменений в DWH-схеме и маршрутах Camel. Цикл согласования занимает недели.

---

## Что даёт событийный подход

### Слабая связанность (loose coupling)

Домен публикует событие в топик Kafka — и больше ничего не знает о подписчиках. Fintech не знает, что Analytics потребляет `fintech.payment.completed`. Healthcare не знает, что Pharmacy слушает `healthcare.prescription.issued`. Добавление нового подписчика не требует изменений в источнике.

**Сравнение:**

| | Camel ESB | Event-Driven |
|---|---|---|
| Добавить нового потребителя события | Изменить маршрут в Camel + деплой | Новый сервис подписывается на топик, источник не меняется |
| Источник упал | Все потребители получают ошибки | Потребители читают из Kafka, когда восстановятся |
| Нагрузка выросла в 10× | Шина становится узким местом | Kafka горизонтально масштабируется, партиции добавляются |

### Независимость доменов

Каждый домен владеет своей БД и своими событиями. Fintech может мигрировать с PostgreSQL на другую СУБД без уведомления Healthcare. Pharmacy может развернуть новую версию сервиса в любой момент — контракт события зафиксирован в Schema Registry.

### Near-real-time аналитика

События из Kafka поступают в Data Lakehouse через Kafka Connect непрерывно. Gold Layer обновляется по мере поступления данных, а не раз в ночь. Бизнес видит в DataLens текущее состояние, а не вчерашнее.

### Масштабирование новых направлений

Подключение Фарма-домена: команда публикует свои события в топики Kafka, подписывается на нужные события Healthcare — и всё. Никаких изменений в DWH, никаких новых маршрутов в Camel. Именно это позволяет выйти на 2–3 новых региона и запустить новые продукты в рамках трёхлетнего горизонта.

### Аудит и воспроизводимость

Kafka хранит полную историю событий (retention настраивается). Любое состояние системы можно восстановить, переиграв события с начала. Это критично для финансового аудита (ЦБ РФ) и медицинских проверок.

### Устойчивость к сбоям

Асинхронная обработка с DLQ: если Pharmacy временно недоступна, событие `PrescriptionIssued` не теряется — оно лежит в топике и будет обработано при восстановлении. В синхронной модели Camel это привело бы к ошибке у пользователя.

---

## Почему нельзя просто улучшить Camel + DWH

1. **SQL Server 2008 снят с поддержки** — патчи безопасности недоступны. Регулятор (152-ФЗ, ЦБ) не примет систему на EOL-платформе.
2. **Camel — синхронный оркестратор**, а не шина событий. Переписать маршруты под async-модель с гарантиями доставки — это фактически замена на Kafka с сохранением только названия.
3. **DWH не масштабируется горизонтально** под write-нагрузку нескольких доменов одновременно. Партиционирование по доменам внутри одного SQL Server не даёт изоляции команд.
4. **Инкрементальное улучшение** монолитного DWH создаёт технический долг быстрее, чем он погашается, — это подтверждает текущая ситуация с многочасовой генерацией отчётов.

---

## Стратегия перехода: не «всё сразу»

Переход не требует одномоментного отказа от Camel/DWH. Антикоррупционный слой (ACL) позволяет:
- DWH продолжает работать в режиме read-only, данные вытягиваются через CDC
- Camel остаётся адаптером для ещё не мигрировавших интеграций
- Каждый новый домен с первого дня работает через Event Bus

Это снижает риск трансформации и позволяет доказать ценность нового подхода на пилоте (Финтех + Пациентский поток) до масштабирования на всю компанию.
