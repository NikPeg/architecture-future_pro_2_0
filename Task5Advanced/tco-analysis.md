# TCO-анализ — «Будущее 2.0»

Горизонт: 3 года. Валюта: млн ₽/год. Все цифры — оценочные, основаны на рыночных данных для компании масштаба «Будущее 2.0» (~500 сотрудников IT, 4 домена).

---

## Допущения

**Текущая архитектура:**
- On-premise ЦОД: 4 физических сервера под DWH + 2 под ESB/BI
- SQL Server 2008 Enterprise — EOL, работает на продлённой поддержке (ESU)
- Power BI Pro: ~60 пользователей
- 3 DBA, обслуживающих DWH
- 5 аналитиков, из которых 60% времени уходит на ручную подготовку данных из-за медленной архитектуры
- Рост данных: +25% в год → растёт время генерации отчётов, растут затраты на аналитиков

**Целевая архитектура:**
- Yandex Cloud (managed-сервисы): Kafka, PostgreSQL, Kubernetes, Object Storage, DataProc
- Команда платформы данных: 4 инженера (новые найм/перераспределение)
- Self-service BI: аналитики тратят 80% времени на анализ, не на подготовку данных
- Миграционные затраты включены в Year 1 и Year 2

- масштабирования без пропорционального роста затрат

## Дополнительные бизнес-выгоды (не включены в TCO)

| Выгода | Оценка эффекта |
|--------|----------------|
| Ускорение time-to-market для новых продуктов | С 6–12 мес. до 1–3 мес. для нового домена |
| Near-real-time аналитика вместо batch overnight | Бизнес-решения на основе актуальных данных |
| Подключение новых партнёров (Фарма, МедТех) без DWH-изменений | Снижение стоимости интеграции на ~3 млн/домен |
| Снижение риска регуляторных штрафов (152-ФЗ, EOL SQL Server) | Потенциальные штрафы до 10+ млн ₽ — исключены |
| Масштабирование на 2–3 новых региона без hardware | Экономия на ЦОД ~5 млн/регион |

## Вывод

Трансформация обходится дороже в первые два года за счёт единовременных миграционных затрат. Однако с Year 3 целевая архитектура становится дешевле в эксплуатации, а с Year 4 даёт устойчивую экономию. При этом ряд критичных рисков (EOL SQL Server, невозможность масштабирования, регуляторные требования) делает сохранение текущей архитектуры нежизнеспособным вне зависимости от стоимости трансформации.
